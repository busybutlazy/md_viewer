# Architectural Decisions

記錄專案中的重要架構決策。輕量級 ADR（Architecture Decision Record）。

## 格式

```
## ADR-NNN: <決策主題>
**日期**：YYYY-MM-DD
**階段**：P<階段>
**狀態**：提議中 / 已採納 / 已取代（被 ADR-XXX 取代）

### 背景
描述為什麼需要做這個決策。

### 考慮過的選項
- 選項 A：...優點...缺點...
- 選項 B：...優點...缺點...

### 決策
選擇 X，因為 Y。

### 後果
這個決策帶來的影響（正面與負面）。
```

### ADR-000: 採納 Next.js App Router 而非 Vite + React Router

**日期**：2026-04-18
**階段**：P1.0
**狀態**：已採納

**背景**：選擇 React 應用的底層框架。

**考慮過的選項**：
- Vite + React Router：輕、快、熟悉度高
- Next.js App Router：檔案系統路由、Server Components、生態整合

**決策**：採用 Next.js 15 App Router。

**原因**：
- 檔案系統路由減少手動維護路由設定
- 若未來需要部署為真正網站、預渲染、SEO，Next.js 省事
- Next.js 能做靜態匯出（`output: 'export'`），階段 3 Electron 也能直接用

**後果**：
- 正面：路由、字型、字串最佳化、image 最佳化都免費
- 負面：比 Vite 重、build 較慢；Server Components 的學習曲線

### ADR-001: 採納雙形態架構（網頁版 + 桌面版同程式碼）

**日期**：2026-04-18
**階段**：P2.0 前規劃
**狀態**：已採納

**背景**：階段 3 要做桌面版，是否保留網頁版？

**考慮過的選項**：
- 只做桌面版：架構簡單
- 雙形態：同一份程式碼，執行時偵測環境

**決策**：採雙形態。

**原因**：
- 網頁版能給臨時使用者（不想裝東西）
- 差異只在檔案存取一層，用 `FileSystemAdapter` 抽象即可
- 未來行動版直接接網頁版

**後果**：
- 正面：最大觸及面、架構乾淨
- 負面：所有檔案系統操作都要走 adapter，初期多一點工
